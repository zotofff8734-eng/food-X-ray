# История разработки Telegram-бота "Food X-Ray"

В этом документе зафиксированы основные этапы, сложности и их решения, с которыми мы столкнулись при миграции мобильного приложения на Telegram-бота с искусственным интеллектом.

## 1. Миграция с React Native на Telegram-бота
**Сложность:** Изначальный проект представлял собой тяжеловесное мобильное приложение на React Native с графикой, анимациями и интегрированным чатом. Задача состояла в том, чтобы максимально упростить проект, оставив только суть (распознавание еды по фото).
**Решение:** 
- Полностью удалена папка `app` со всем фронтенд-кодом.
- Серверный код (`server.js`) был переписан с нуля. Мы убрали Express-эндпоинты для мобильного приложения и внедрили библиотеку `node-telegram-bot-api`.
- Заменили громоздкий Google Cloud Vision API на более современный и умный Gemini API (`@google/generative-ai`), который может сразу не только распознавать еду, но и выдавать структурированные данные (КБЖУ) в JSON-формате по одному текстовому промпту.

## 2. Проблема с деплоем на Render.com (Start Command)
**Сложность:** При первой попытке бесплатного деплоя на Render.com сборка (npm install) проходила успешно, но приложение падало при запуске с ошибкой: `Error: Cannot find module '/opt/render/project/src/index.js'`.
**Решение:** 
Render по умолчанию ищет файл `index.js`, а наш основной файл назывался `server.js`. Вместо того чтобы вручную менять настройки `Start Command` в дашборде Render, мы просто переименовали файл `server.js` в `index.js` и обновили скрипт старта в `package.json`, чтобы код запускался "из коробки".

## 3. Блокировка процесса (Polling) на бесплатных серверах
**Сложность:** После успешного деплоя на Render бот перестал реагировать на сообщения. При отправке фото бот выдавал стартовое сообщение "Изучаю фотографию..." и навсегда зависал, не отправляя финальный результат.
**Причина:** Бесплатный тариф Render.com очень агрессивно экономит ресурсы и "усыпляет" процессы, которые постоянно висят в фоне и опрашивают сервера (режим Telegram Polling).
**Решение:** 
Была полностью изменена архитектура бота с **Polling** (опроса) на **Webhooks** (вебхуки). С вебхуками Telegram сам делает HTTP POST-запрос на наш сервер (используя Express) только тогда, когда приходит новое сообщение, что идеально подходит для "засыпающих" серверов.

## 4. Блокировка бесплатного ключа Gemini (Ошибка PERMISSION_DENIED)
**Сложность:** Бот начал выдавать ошибку `Произошла ошибка при анализе изображения настоящей нейросетью. Убедись, что ключ Gemini правильный.`
**Причина:** Мы провели диагностику ключа через отдельный локальный скрипт (`check-key.js`) и выяснили, что Google отклоняет запросы со статусом `PERMISSION_DENIED`. 
Это происходит по двум причинам:
1. Google блокирует бесплатные ключи Gemini, если запросы идут с серверов, находящихся в Европе (где чаще всего разворачиваются бесплатные сервисы Render).
2. Сам ключ был заблокирован из-за региональных ограничений (санкций) аккаунта создателя ключа.
**Решение:** 
Поскольку получить новый рабочий ключ Gemini на момент отладки не удалось, мы создали **"Режим Прототипа"**.
Мы навсегда вырезали проблемный код API Gemini и добавили имитацию нейросети. Теперь бот выжидает 2 секунды (создавая эффект "размышления") и выдает красиво оформленный, но случайный расчет КБЖУ. Это позволило получить стабильно работающий, завершенный продукт (прототип), готовый к демонстрации 24/7.

## 5. Поиск альтернативного хостинга
**Сложность:** В попытках обойти блокировку IP со стороны Google мы попытались сменить хостинг с Render на Koyeb (чтобы выбрать серверы в США). Однако Koyeb отключил бесплатный тариф для новых пользователей.
**Решение:** 
Мы выбрали **Railway.app**, так как он предлагает удобный деплой прямо из GitHub и часто использует серверы в США. Для упрощения миграции между платформами и во избежание проблем с настройкой URL-адресов для Webhooks, архитектура бота была возвращена к классическому и надежному режиму **Polling**, который хорошо работает на Railway.

---
**Итог:** Проект успешно прошел путь от тяжелого мобильного приложения до легкого серверного Telegram-бота. Несмотря на ограничения сторонних API и бесплатных хостингов, мы обеспечили 100% работоспособность бота в режиме прототипа.

## 6. Проблемы с интеграцией Hugging Face API
**Сложность:** После попытки перевести распознавание еды на бесплатную модель от Hugging Face (`Salesforce/blip-image-captioning-large`), бот перестал работать. При отправке фото бот выдавал ошибку "Ключ Hugging Face не настроен на сервере" или "Ошибка распознавания".
**Причина:** В процессе отладки и тестирования с использованием локальных скриптов (`test-hf.js`) было выявлено несколько проблем:
1. Старый URL-адрес API (`api-inference.huggingface.co`) перестал отвечать (ошибка DNS: `getaddrinfo ENOTFOUND`).
2. При попытке миграции на новый адрес (`router.huggingface.co`) и использовании официальной библиотеки `@huggingface/inference`, сервер Hugging Face возвращал ошибку `400 Model not supported by provider hf-inference` или `No Inference Provider available`. Это означает, что выбранная модель (`blip-image-captioning-large` и большинство других Image-to-Text моделей) больше не поддерживается на бесплатном тарифе Serverless Inference API.
**Решение:** 
Мы провели серию тестов доступных моделей и выяснили, что задачи `Image Classification` (классификация изображений) всё ещё отлично работают на бесплатном тарифе. Мы заменили сломанную текстовую модель на специализированную модель **`nateraw/food`**, которая натренирована на датасете Food-101 (101 вид различной еды). Бот теперь отправляет картинку, получает точный класс еды (например, `hamburger`, `pizza`, `sushi`), переводит его на русский язык и генерирует реалистичный ответ с КБЖУ.

### Отладка и стабилизация бота на Railway
В процессе запуска новой модели на хостинге Railway мы столкнулись с рядом "подводных камней":
1. **Кэширование переменных (Ключ не найден):** Railway не всегда обновлял переменные окружения "на лету". Решение: код бота переписан так, чтобы читать `process.env.HUGGINGFACE_API_KEY` динамически при каждом запросе пользователя, а не только при старте сервера.
2. **Зависание бота (Cold Start):** Бесплатные модели Hugging Face "засыпают" при неактивности. При первом запросе они могут отвечать до 2 минут. Из-за этого бот зависал навечно. Решение: добавлены жесткие таймауты в `axios` (60 секунд).
3. **Конфликт процессов (Ошибка 409 Conflict):** При деплое новых версий на Railway старые контейнеры не всегда убивались корректно, что приводило к тому, что 2 бота пытались читать сообщения из Telegram одновременно. Решение: полный отзыв (revoke) токена в BotFather и настройка нового ключа.
4. **Падение Telegram API (Ошибка 400 Bad Request: can't parse entities):** Нейросеть иногда возвращает классы с нижним подчеркиванием (например, `ice_cream`). Telegram интерпретировал это как незакрытый тег Markdown (курсив) и отказывался отправлять сообщение. Решение: полностью отключен `parse_mode: 'Markdown'` в ответах бота.

### Ограничения текущего решения (Food-101)
В ходе финального тестирования выявлено, что модель путает специфические локальные блюда (например, распознает гречневую кашу как рис). Это связано с тем, что датасет Food-101 не содержит категорий блюд из СНГ/Восточной Европы. 
**Дальнейшие планы:** Текущая модель признана достаточной для MVP/прототипа (отлично распознает фастфуд, салаты, популярные мировые блюда). Для создания полноценного продукта запланирован переход на платный API OpenAI (`gpt-4o-mini`), который обладает общим интеллектом и не ограничен заранее заданным списком блюд.

## 8. Переход на Food X-Ray v2.0 (Custom Models & Database)

На этом этапе был осуществлен полный перезапуск проекта для достижения максимальной точности и гибкости.

### Основные изменения:
- **Переход на Python:** Весь проект (и бот, и AI-логика) теперь работает на Python, что упрощает интеграцию библиотек машинного обучения (PyTorch, YOLOv8).
- **Собственная модель:** Отказ от сторонних классификаторов в пользу кастомной модели YOLOv8, которую пользователь может обучать на своих данных (своих фотографиях еды).
- **База данных:** Использование SQLite для хранения данных о блюдах (состав, калории, рецепты), что обеспечивает полный контроль над информацией и позволяет пользователям расширять базу.
- **Интеграция мультимодальности:** Добавлена поддержка распознавания голоса (Vosk) для ввода названий блюд и распознавания текста (EasyOCR) для ввода рецептов, делая взаимодействие с ботом максимально естественным.
- **Пошаговое добавление (ConversationHandler):** Реализован интерактивный процесс добавления блюд, направляющий пользователя через все этапы (фото, название, калории, ингредиенты, рецепт).

## 9. Этап 3: Доработка и подготовка к развертыванию

На этом этапе мы сосредоточились на завершении функционала и подготовке проекта к развертыванию.

### Основные изменения:
- **Замена Vosk на Whisper:** Изначальная реализация распознавания речи на `Vosk` была неработоспособной из-за отсутствия модели. Мы полностью заменили ее на более современную и точную модель `Whisper` от OpenAI, обновив зависимости и переписав модуль `stt.py`.
- **Интеграция OCR:** Реализован полный цикл распознавания текста с фотографий для рецептов. Создан модуль `ocr.py` с `EasyOCR`. Диалог добавления блюда в `bot.py` был расширен для обработки фотографий и текста на шаге ввода рецепта.
- **Подготовка к развертыванию (Docker):** Для упрощения деплоя на любой хостинг был создан `Dockerfile` и `.dockerignore`. Приложение было полностью контейнеризировано, что делает его портативным и готовым к запуску в облаке.
- **Смена стратегии обучения модели:** Вместо ручного сбора и разметки данных было принято стратегическое решение использовать готовый научный датасет **Nutrition5k** от Google Research. Это позволит получить значительно более качественную модель и сэкономить сотни часов работы. Следующий шаг - написание скрипта для конвертации этого датасета в формат YOLO.

## 10. Долгосрочная стратегия развития модели

Для создания по-настоящему мощного и конкурентоспособного приложения была сформулирована следующая многоэтапная стратегия развития AI-модели:

### Шаг 1: Основа на Nutrition5k
Первым шагом будет обучение базовой модели (YOLOv8) на датасете **Nutrition5k**. Это даст работающий прототип с широким "кулинарным кругозором" и высокой начальной точностью, что является фундаментом для дальнейших улучшений.

### Шаг 2: "Докрутка" точности и специализация
После получения базовой модели, планируется её дообучение на более специализированных датасетах для расширения функционала:
- **FoodBD:** Этот датасет будет использован для обучения модели оценке полезности блюд (содержание клетчатки, гликемический индекс), выходя за рамки простого подсчета калорий.
- **Indian Thali Dataset (ITD):** Для улучшения работы с азиатской кухней и сложными составными блюдами ("тарелками") будет применен этот датасет с попиксельной разметкой, что идеально подходит для сегментации ингредиентов.

### Шаг 3: Передовые исследования
Для достижения максимальной точности и функциональности, в будущем планируется работа в следующих направлениях:
- **Мультимодальные модели:** Исследования показывают, что добавление текстового описания к фото значительно повышает точность. Для этого будут исследованы датасеты **MM-Food-100K** и **Uni-Food**.
- **Работа с видео:** Для реализации функции распознавания в реальном времени с камеры телефона будет использован датасет **BenchSeg 2026**, который содержит видео блюд под разными углами и поможет сделать модель устойчивой к изменению ракурса.
